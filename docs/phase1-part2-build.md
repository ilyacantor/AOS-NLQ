# WS2.1: Cross-Sell Propensity Engine

## START CONDITION: Immediately. Customer data with cross-sell signal fields already exists from Part 1.

## Overview

Match customer firmographic + behavioral attributes against buyer profiles per product/service line. Two pathways: Meridian advisory → Cascadia's BPM clients, and Cascadia BPM → Meridian's consulting clients. Output: propensity-scored candidates with estimated ACV that feeds the EBITDA bridge.

## The matching logic

### Meridian's service catalog (what can be sold to Cascadia's clients):
```
Practice Areas:
- Strategy: typical buyer = CEO/CFO, $250K-$2M ACV, industries: all, trigger: M&A, market shift, board mandate
- Operations: typical buyer = COO/VP Ops, $300K-$1.5M ACV, trigger: post-merger integration, process redesign
- Technology: typical buyer = CTO/CIO, $200K-$1.2M ACV, trigger: legacy modernization, scaling pain
- Risk & Compliance: typical buyer = CRO/CCO, $250K-$2.5M ACV, trigger: regulatory pressure, audit findings
- Digital/AI: typical buyer = CDO/CTO, $300K-$2M ACV, trigger: automation mandate, data strategy
- Commercial: typical buyer = CRO/CMO, $150K-$800K ACV, trigger: GTM redesign, channel expansion
```

### Cascadia's service catalog (what can be sold to Meridian's clients):
```
Service Lines:
- F&A Outsourcing: typical buyer = CFO/VP Finance, $500K-$5M ACV, trigger: post-transformation handoff, cost reduction mandate
- CX Management: typical buyer = VP CX/VP Shared Services, $800K-$8M ACV, trigger: CX fragmentation, multi-channel pain
- Data & Analytics: typical buyer = CDO/VP Analytics, $300K-$2M ACV, trigger: data ops gap, reporting burden
- Industry Process Solutions: typical buyer = COO/VP Ops, $1M-$10M ACV, trigger: regulatory ops, industry-specific process complexity
```

### Propensity scoring model:

For each non-overlapping customer in one entity, score against each product/service in the other entity. Scoring dimensions:

**Industry match (0-25 points):** Does the customer's industry align with the product's typical buyer industries? Exact match = 25. Adjacent = 15. Weak = 5.

**Company size match (0-20 points):** Employee count and revenue band vs. the product's sweet spot. Within range = 20. Close = 12. Far = 4.

**Behavioral signals (0-30 points):** These are the fields generated in Part 1 on customer records.
- For M advisory → C clients: process complexity (0-10), regulatory burden (0-5), recent M&A (0-5), growth rate (0-5), escalation history (0-5)
- For C BPM → M clients: manual process count (0-10), outsourcing readiness (0-5), transformation maturity (0-5), engagement recency (0-5), expressed interest (0-5)

**Engagement fit (0-15 points):** Does the customer's current spending pattern suggest need?
- For C clients: high BPO spend + complexity = advisory need. Multi-domain outsourcing = transformation candidate.
- For M clients: recent project completion + identified ops gaps = BPM candidate. High manual process count = outsourcing fit.

**Relationship strength (0-10 points):** How deep is the existing relationship?
- Years as client (longer = warmer intro)
- Contract size (larger = more trust)
- Multi-product (already buying multiple things = cross-buy propensity)

**Total: 0-100.** Thresholds: >80 = high confidence, 60-80 = medium, 40-60 = low, <40 = not a candidate.

### Estimated ACV:

For each candidate above 40, estimate ACV based on:
- Product's typical ACV range
- Customer's size (larger companies = higher ACV)
- Behavioral intensity (more signals = closer to top of range)
- Comparable customer analysis: find existing customers of the target product that look most like this candidate, use their ACV as a reference point

# WS2.2: EBITDA Bridge Engine

## START CONDITION: After WS2.1 (cross-sell feeds into revenue synergy line).

## Overview

Takes the unified trial balance from Part 1's COFA engine + a set of adjustment items and produces the EBITDA bridge as a FORMULA, not a static table. Each adjustment is a parameterized function that the what-if engine (WS2.3) can modify.

## Bridge structure

```
Reported EBITDA (from combining P&L)
  + Entity-Level Adjustments
    + Above-market exec compensation (M: $18M, C: $8M)
    + Non-recurring: M litigation reserve ($22M)
    + Non-recurring: C delivery center closure ($35M)
    + Transaction costs — this deal (M: $15M, C: $8M)
    + Related-party leases — C founder properties ($4M)
    + Run-rate: new contracts not at full revenue (M: $28M, C: $15M)
    + Utilization normalization — M Q2-Q3 dip ($45M)
    + Offshore labor mix shift — C in progress ($12M)
    + C attrition normalization 18% → 15% ($18M)
  = Entity-Level Adjusted EBITDA

  + Combination Synergies
    + Bench optimization — consulting ($80-120M)
    + Bench optimization — delivery ($25-45M)
    + Corporate function consolidation ($45-70M)
    + Vendor consolidation ($15-25M)
    + Technology redundancy elimination ($8-12M)
    + Cross-sell revenue contribution — net margin ($35-65M) ← from WS2.1
    - Integration costs Year 1 ($85-120M)
    - Retention packages ($20-30M)
  = Pro Forma Adjusted EBITDA
```

### Each adjustment is a formula:

```python
class Adjustment:
    name: str
    category: str        # "normalization", "one_time", "run_rate", "cost_synergy", "revenue_synergy", "dis_synergy"
    entity: str          # "meridian", "cascadia", "combined"
    confidence: str      # "high", "medium", "low"
    
    # The formula — this is what makes it parameterizable for what-if
    base_amount: float              # the default/expected amount
    sensitivity_lever: str | None   # which slider controls this (e.g., "bench_rate", "cross_sell_capture")
    sensitivity_formula: callable   # f(lever_value) → adjusted amount
    
    support_reference: str          # what data/document supports this adjustment
    rationale: str                  # plain language explanation
```

### Key formulas:

**Bench optimization (consulting):**
```
base: $100M (midpoint)
lever: "bench_cross_deploy_rate" (default 15%, range 0-50%)
formula: bench_consultants × bench_cost_per_month × 12 × deploy_rate × (billing_realization - 1)
  where: bench_consultants = 4,500, bench_cost_per_month = $15,100, billing_realization = ratio of billable revenue to loaded cost
sensitivity: ~$14M per point of deploy rate
```

**Bench optimization (delivery):**
```
base: $35M (midpoint)
lever: "bench_cross_deploy_rate" (same lever, different formula)
formula: bench_delivery_ftes × bench_cost_per_month × 12 × deploy_rate
  where: bench_delivery_ftes = 4,200, bench_cost_per_month = $1,800
```

**Corporate function consolidation:**
```
base: $57M (midpoint)
lever: "corporate_hc_reduction_pct" (default 20%, range 0-40%)
formula: overlapping_corporate_hc × avg_corporate_comp × reduction_pct
  where: overlapping_corporate_hc from Part 1 entity resolution, avg_corporate_comp = blended
data_source: People overlap report from Part 1
```

**Vendor consolidation:**
```
base: $20M (midpoint)
lever: none (static — based on identified overlap from Part 1)
formula: sum of estimated savings per overlapping vendor
data_source: Vendor overlap report from Part 1
```

**Cross-sell revenue contribution:**
```
base: $50M at 50% capture (midpoint)
lever: "cross_sell_capture_rate" (default 50%, range 0-100%)
second lever: "cross_sell_ramp_months" (default 18, range 6-36)
formula: total_pipeline_acv × capture_rate × net_margin_pct × ramp_factor(months)
  where: total_pipeline_acv from WS2.1, net_margin_pct = 30%, ramp_factor = months_completed / ramp_months
data_source: Cross-sell pipeline from WS2.1
```

**Integration costs:**
```
base: $100M Year 1
lever: "integration_cost" (default $100M, range $50-150M)
formula: direct input (slider sets the value)
applies: Year 1 only. Year 2+ = $0.
```

# WS2.3: What-If / Sensitivity Engine

## START CONDITION: After WS2.2 (EBITDA bridge provides the formula).

## Overview

Makes the EBITDA bridge interactive. 10 slider inputs, each adjusting specific bridge lines. Output updates in real time: adjusted P&L, EBITDA bridge, and EV impact.

## The 10 levers

```
1. M Utilization Rate
   Range: 72-85%    Default: 78%    Impact: ~$62M per point
   Affects: Revenue (utilization × consultants × billing rate) + bench cost (inverse of utilization)
   
2. M Bench Rate
   Range: 10-22%    Default: 18%    Impact: ~$18M per point
   Affects: Bench COGS line directly

3. C Offshore Mix
   Range: 50-80%    Default: 60%    Impact: ~$6M per point
   Affects: Labor cost blend (offshore is cheaper)

4. C Attrition Rate
   Range: 10-22%    Default: 18%    Impact: ~$4M per point
   Affects: Recruiting + training cost, run-rate normalization

5. Cross-Sell Capture Rate
   Range: 0-100%    Default: 50%    Impact: ~$1.3M per point
   Affects: Revenue synergy line in EBITDA bridge
   Data source: WS2.1 pipeline

6. Cross-Sell Ramp (months to full)
   Range: 6-36       Default: 18     Impact: varies by year
   Affects: When revenue synergy hits the P&L (Year 1 vs Year 2 vs Year 3)

7. Corporate HC Reduction %
   Range: 0-40%      Default: 20%    Impact: ~$2.4M per point
   Affects: G&A consolidation synergy
   Data source: Part 1 people overlap

8. Bench Cross-Deploy Rate
   Range: 0-50%      Default: 15%    Impact: ~$14M per point
   Affects: Both bench optimization lines (consulting + delivery)

9. Integration Cost
   Range: $50-150M   Default: $100M  Impact: direct
   Affects: Year 1 dis-synergy line

10. COFA Harmonization Choice
    Type: per-line toggle (not a slider)
    Options per conflict: "Harmonize to M" | "Harmonize to C" | "Conservative" | "As-reported"
    Affects: Revenue reclassifications, OpEx reclassifications in the combining P&L
    Data source: Part 1 conflict register
```

## Engine design

The what-if engine is a function:

```python
def compute_what_if(
    levers: dict,                    # current slider values
    ebitda_bridge: EBITDABridge,     # from WS2.2
    combining_pl: CombiningPL,       # from Part 1
    cross_sell_pipeline: Pipeline,   # from WS2.1
    cofa_conflicts: ConflictRegister # from Part 1
) -> WhatIfResult:
    
    # 1. Apply COFA harmonization toggles to the combining P&L
    adjusted_pl = apply_cofa_choices(combining_pl, levers["cofa_harmonization"], cofa_conflicts)
    
    # 2. Recompute reported EBITDA from the adjusted P&L
    reported_ebitda = compute_ebitda(adjusted_pl)
    
    # 3. Apply entity-level adjustments (mostly static, but utilization + bench rate affect some)
    entity_adjusted = apply_entity_adjustments(reported_ebitda, levers, ebitda_bridge.entity_adjustments)
    
    # 4. Compute combination synergies using lever values
    synergies = compute_synergies(levers, ebitda_bridge.combination_synergies, cross_sell_pipeline)
    
    # 5. Produce the result
    return WhatIfResult(
        adjusted_pl=adjusted_pl,
        ebitda_bridge=recomputed_bridge,
        pro_forma_ebitda=entity_adjusted + synergies,
        ev_impact=pro_forma * levers.get("multiple", 12.5)
    )
```

### Preset scenarios (pre-computed what-if combinations):

```
Base (as-reported, naive sum): all levers at "no adjustment" positions
Harmonize to M: COFA = all to M, utilization = current, bench = current
Harmonize to C: COFA = all to C
Conservative (Big 4 QofE): lowest reasonable synergies, highest dis-synergies, conservative COFA choices
Aggressive (full capture, Year 3): highest synergies, lowest dis-synergies, full cross-sell, max bench deploy
```

Users can start from a preset and then adjust individual levers.

### Validation:
- Moving a single lever updates the EBITDA bridge correctly (only the affected lines change)
- Cross-sell capture at 0% = $0 revenue synergy. At 100% = full pipeline ACV × margin.
- Bench cross-deploy at 0% = $0 bench synergy. At 50% = max bench optimization.
- COFA toggles change the combining P&L adjustment column.
- Preset scenarios produce expected values (match ground truth for each preset).
- EV impact = EBITDA × multiple at all times.
- The what-if engine runs in <2 seconds (it's recomputing a formula, not re-querying the database).

# WS2.5: Maestra Engagement Lifecycle

## START CONDITION: Can start design in parallel with other Part 2 workstreams. Full build after dashboards exist.

## Overview

Extend the existing onboarding agent spec into a full Convergence engagement manager. Maestra handles three phases: scoping, execution, and ongoing management.

### Phase 1: Scoping (onboarding)
- **Object scoping conversation:** Which of the five defaults are in scope? (All by default.) Any domain-specific extensions needed?
- **Entity profile capture:** For each entity — name, business type, scale, key systems, fiscal year, primary contacts.
- **System access coordination:** What systems can we connect to? AOD needs credentials or API access for both entities.
- **Stakeholder mapping:** Who are the decision-makers? CFO, CRO, COO, CTO, CHRO — who is the primary sponsor?
- **Timeline and priority:** Due diligence timeline? Which reconciliation objects are highest priority?

Output: Engagement configuration that drives the pipeline. Which objects to reconcile, which systems to discover, who to notify at each milestone.

### Phase 2: Execution (status tracking)
- **Status per reconciliation object:** Financial Statements: complete/in-progress/pending. Customers: complete. Vendors: in-progress. People: pending. IT: complete.
- **Conflict resolution tracking:** For each conflict in the register — status (identified/under-review/resolved), assigned stakeholder, resolution decision.
- **Milestone notifications:** "COFA unification complete. 6 conflicts identified. CFO review needed."
- **Deliverable generation:** "Generate the combining P&L report" → triggers report generation from the engines.
- **Follow-up coordination:** "The revenue gross-up conflict needs CFO input. Schedule a 30-min call."

### Phase 3: Ongoing management
- **Synergy scorecard:** Track actuals vs. identified synergy opportunities over time.
- **Drift alerts:** "Cascadia changed their depreciation method. The COFA mapping for account 5400 may be affected."
- **New acquisition onboarding:** "A third entity is being acquired. Same process, same engine — starting scoping."

### Implementation:

Maestra is a conversation agent (LLM-powered) with structured state. The state includes:
- Engagement config (entities, objects in scope, timeline)
- Status per object and per conflict
- Stakeholder contact info and assignments
- History of decisions and actions

Maestra's conversation interface lives in the portal as a chat panel or dedicated tab. She can reference any report, any data point, any conflict — and link to the relevant portal view.

### Validation:
- Maestra can run a scoping conversation that produces a valid engagement config
- Maestra can report status per reconciliation object
- Maestra can describe a specific conflict from the register in plain language
- Maestra can trigger report generation
- Maestra's state persists across sessions

- Do NOT modify Phase 0 or Part 1 test suites. Add new test groups only.

- Do NOT make the what-if engine re-query the database on every slider move. It's a formula evaluation — preload the data, compute in memory.
