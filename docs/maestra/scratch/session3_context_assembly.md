# CC Session 3: Constitution + Context Assembly

## Context
We are building Maestra, the AI engagement lead for AOS Convergence (M&A). This session builds the core brain: loading the constitution, assembling context from all three sources, calling the LLM, and returning Maestra's response.

Sessions 1 (module status endpoints) and 2 (engagement API) must be complete.

Read tests/HARNESS_RULES.md before making any changes. All 16 rules apply.

## What to build

### Step 1: Place constitution documents

Copy `constitution/base.md` and `constitution/convergence.md` into `src/maestra/constitution/` in the NLQ repo. These are provided as files in this session.

### Step 2: Build context assembly

Create `src/maestra/context.js`:

```javascript
async function assembleContext(customerId, userMessage, sessionId) {
  // 1. Load constitution
  //    - Always load base.md
  //    - Load scenario-specific constitution based on engagement.scenario_type
  //    For convergence: load convergence.md
  
  // 2. Load engagement state
  //    - getEngagement(customerId) -> customer info, deal phase, entities
  //    - getRecentMemory(customerId, 5) -> last 5 interactions for continuity
  //    - customer_playbooks -> vocabulary, priorities, systems
  
  // 3. Load module state
  //    - For each module (aod, aam, farm, dcl): getModuleState(module, customerId)
  //    - If cache is stale (>5 min) or missing, fetch live from module status endpoints
  //    - Update cache with fresh data
  
  // 4. Assemble the prompt
  //    System message = constitution (base + scenario)
  //    + "Current engagement state:" + JSON summary of engagement
  //    + "Recent conversation context:" + formatted session memory
  //    + "Live platform state:" + JSON from each module
  //    + "Customer context:" + playbook (vocabulary, priorities, systems)
  //    
  //    User message = the customer's message as-is

  // 5. Call LLM
  //    - Use Claude API (same client NLQ uses)
  //    - Model: claude-sonnet-4-6 (hardcoded for now, parameterize for future routing)
  //    - Temperature: 0.3 (factual, not creative)
  //    - max_tokens: 2000
  
  // 6. Parse response
  //    - Check if response contains a JSON action block (```json { "action": ... } ```)
  //    - If yes: extract the action, return it separately
  //    - Return: { text: string, action: object|null, usage: { input_tokens, output_tokens } }
  
  // 7. Log interaction
  //    - Hash the assembled prompt (for future semantic cache matching)
  //    - Log to interaction_log: customer_id, session_id, input_hash, model, tokens, latency
  
  // 8. Update engagement state
  //    - Update last_interaction_at
  //    - Add session_memory entry summarizing this interaction

  return { text, action, usage, latencyMs };
}
```

### Key implementation details

**Constitution loading:** Read the .md files from disk at startup (or on first call, cached in memory). Do NOT read from disk on every request.

**Module state freshness:** The module_state_cache table has an updated_at field. If a cached entry is older than 5 minutes, fetch fresh from the module's GET /maestra/status endpoint (from Session 1) and update the cache. If the module is unreachable, use stale cache and note it in the context: "Note: [module] status is from [timestamp], may be stale."

**Action block parsing:** Maestra's response may contain a fenced JSON block with an action. Use a regex to extract it:
```
/```json\s*(\{[\s\S]*?"action"[\s\S]*?\})\s*```/
```
Remove the action block from the display text (customer shouldn't see raw JSON).

**Session memory summary:** Do not store the full LLM response in session_memory. Store a 1-2 sentence summary of what was discussed and what action (if any) was taken. The summary can be generated by the LLM as part of the response, or extracted heuristically.

**Prompt size guard:** Before calling the LLM, count approximate tokens (chars / 4 as rough estimate). If the assembled prompt exceeds 15,000 tokens, truncate session memory (oldest first) until it fits. Log a warning when truncation occurs.

## Harness

Create tests/maestra/context_assembly.test.js

Tests MUST hit the real assembleContext function with real Supabase and real LLM calls (not mocked).

Test cases (use the Meridian/Cascadia demo customer):

1. **Status check:** Send "Where are we with the deal?" 
   - Assert response.text is non-empty
   - Assert response.text mentions both "Meridian" and "Cascadia" (entity clarity rule)
   - Assert response.action is null (status check, no action needed)
   - Assert response.usage.input_tokens > 0 and output_tokens > 0
   - Assert response.latencyMs < 15000

2. **Module-specific question:** Send "What did AOD find for the target?"
   - Assert response.text mentions "Cascadia" (the target)
   - Assert response.text references discovery findings (systems, counts, etc.)
   
3. **Action request (read):** Send "Show me the overlap report"
   - Assert response.action is not null
   - Assert response.action.type === "read"
   - Assert response.action.module === "nlq"
   
4. **Action request (write):** Send "Re-run discovery for Cascadia's Salesforce"
   - Assert response.action is not null
   - Assert response.action.type === "write"
   - Assert response.action.module === "aod"
   
5. **Unknown question:** Send "What's the weather like?"
   - Assert response.text does NOT hallucinate platform data
   - Assert response.text indicates this is outside Maestra's scope
   
6. **Entity ambiguity:** Send "What's the revenue?"
   - Assert response.text asks which entity (Meridian, Cascadia, or combined)
   
7. **Session continuity:** Send two messages in sequence with the same sessionId
   - Assert second response references context from the first
   
8. **Interaction logging:** After any test, query maestra.interaction_log
   - Assert a row was created with correct customer_id, model_used, token counts

9. **Engagement state update:** After any test, query maestra.customer_engagements
   - Assert last_interaction_at was updated

10. **Constitution loading:** Assert that the assembled prompt contains text from both base.md and convergence.md (check for a distinctive phrase from each)

Tests MUST NOT:
- Mock the LLM (use real Claude API calls)
- Assert exact response text (LLM responses vary; assert semantic properties)
- Skip or xfail
- Modify constitution documents
- Exceed 30 seconds per test (LLM calls should be under 15s)

Run ALL existing NLQ tests + Sessions 1-3 Maestra tests. Zero regressions. 100% pass or not done.
